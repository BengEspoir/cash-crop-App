# AgriculNet Deployment Guide

This project is split into a Next.js frontend in `client/`, an Express API in `server/`, and Supabase-backed data storage. The lowest-friction hosting path is Vercel for the frontend, Render for the backend, and Supabase for PostgreSQL/auth-adjacent storage.

## 1. Recommended Hosting Architecture

- **Frontend:** Vercel, connected to GitHub for automatic production and preview deployments.
- **Backend:** Render Web Service as the primary low-cost option for the Express server.
- **Backend alternative:** Railway if you prefer its developer workflow and accept its trial/credit model.
- **Database:** Supabase free tier for PostgreSQL and storage-backed application data.
- **Media:** Cloudinary if image upload features are enabled in production.

## 2. Free Subdomain Options

- Vercel provides a free `*.vercel.app` frontend URL.
- Render provides a free `*.onrender.com` backend URL.
- Railway provides generated service domains while the project has available credit.
- Supabase provides a project URL for database/API access.

## 3. Custom Domain Reality

A custom domain is usually not free because you must buy the domain from a registrar. Vercel and Render can attach a custom domain at no extra platform cost, but the domain itself must be owned by you. For a school/demo project, use the free Vercel and Render subdomains first.

## 4. Frontend Deployment On Vercel

1. Push the repository to GitHub.
2. Create a Vercel project from the GitHub repository.
3. Set the Vercel root directory to `client`.
4. Use the default build command:
   ```bash
   npm run build
   ```
5. Use the default output generated by Next.js.
6. Add environment variables from `client/.env.local.example`, replacing local URLs with production URLs.
7. Set `NEXT_PUBLIC_API_URL` to the deployed backend API URL, ending with `/api/v1`.

Example:
```env
NEXT_PUBLIC_API_URL=https://your-render-service.onrender.com/api/v1
NEXT_PUBLIC_APP_NAME=AgriculNet
NEXT_PUBLIC_APP_URL=https://your-vercel-app.vercel.app
NEXT_PUBLIC_SUPABASE_URL=your-supabase-url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-supabase-anon-key
NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME=your-cloudinary-cloud-name
NEXT_PUBLIC_CLOUDINARY_UPLOAD_PRESET=agriculnet_unsigned
```

## 5. Backend Deployment On Render

1. Create a Render Web Service from the GitHub repository.
2. Set the root directory to `server`.
3. Use Node as the runtime.
4. Use this build command:
   ```bash
   npm install
   ```
5. Use this start command:
   ```bash
   npm start
   ```
6. Add environment variables from `server/.env.example`.
7. Set `NODE_ENV=production`.
8. Set `CLIENT_URL`, `EMAIL_VERIFY_URL`, and `PASSWORD_RESET_URL` to the Vercel frontend domain.

Production URL examples:
```env
CLIENT_URL=https://your-vercel-app.vercel.app
EMAIL_VERIFY_URL=https://your-vercel-app.vercel.app/verify-email
PASSWORD_RESET_URL=https://your-vercel-app.vercel.app/reset-password
```

## 6. Backend Alternative On Railway

1. Create a Railway project from the GitHub repository.
2. Select the `server` folder as the service root if prompted.
3. Configure the same environment variables used for Render.
4. Use `npm install` as the install/build step and `npm start` as the start command.
5. Watch credit limits carefully; Railway is convenient but not always permanently free.

## 7. Environment Variables

Never commit real `.env` files. Keep real values only in Vercel, Render/Railway, Supabase, and your local machine.

Required backend groups:
- Supabase: `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`
- JWT: `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`
- Email: SMTP credentials or a production email provider
- SMS: Africa's Talking or Twilio credentials
- Client URLs: `CLIENT_URL`, `EMAIL_VERIFY_URL`, `PASSWORD_RESET_URL`
- Admin route: set a strong `ADMIN_ROUTE_SECRET`

Required frontend groups:
- API URL: `NEXT_PUBLIC_API_URL`
- Supabase public values: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- App URL/name: `NEXT_PUBLIC_APP_URL`, `NEXT_PUBLIC_APP_NAME`
- Cloudinary public values if uploads are enabled

## 8. Database Setup With Supabase

1. Create a Supabase project.
2. Run the SQL migrations in `server/database/migrations` in numeric order.
3. Run seed files only when you intentionally need demo/admin data.
4. Store the service-role key only in backend hosting environment variables.
5. Do not expose the service-role key in Vercel or client-side code.

## 9. GitHub Auto-Deployment Workflow

- Vercel deploys previews for pull requests and production for the configured production branch.
- Render can auto-deploy the backend when changes land on the selected branch.
- Keep frontend and backend environment variables synchronized when deployment domains change.
- After backend deployment, update `NEXT_PUBLIC_API_URL` in Vercel and redeploy the frontend.

## 10. Manual Deployment Fallback

Frontend:
```bash
cd client
npm install
npm run build
npm run start
```

Backend:
```bash
cd server
npm install
npm start
```

Use this only for temporary VPS/manual hosting. Managed GitHub deployments are safer for the school/demo lifecycle.

## 11. Production Checklist

- No `.env` files are committed.
- `NODE_ENV=production` is set on the backend.
- Backend CORS `CLIENT_URL` matches the Vercel URL.
- `NEXT_PUBLIC_API_URL` points to the deployed backend and includes `/api/v1`.
- Supabase migrations have run successfully.
- JWT secrets are strong random values.
- Email verification works through production SMTP/provider credentials.
- SMS verification works with live provider credentials and sufficient balance.
- `AT_SENDER_ID` is blank unless the sender ID is approved.
- Cloudinary upload preset is configured if image uploads are enabled.
- Frontend build passes.
- Backend starts successfully on the hosting platform.
